漏洞等级：
分为三个等级：高危、中危、低危。

以下评级会根据漏洞的可利用性和影响进行调整，影响因素包含厂商权重、网站流量、攻击复杂度、对被测目标的影响、漏洞利用的权限需求、影响范围、机密性以及完整性等。

# 强制执行协议

本文件是授权测试、漏洞归档、阶段性报告、正式报告和赏金提交的唯一评级依据。发现任何新漏洞、风险候选，或准备把线索从候选升级为有效漏洞时，必须先执行本协议。

未完成本协议的条目：

- 不得标记为 `confirmed`。
- 不得定为中危或高危成果。
- 不得进入正式报告或阶段报告的有效漏洞章节。
- 只能标记为信息项、低危观察、候选、受限未验证或 `blocked_need_material`。

每条漏洞/风险必须在 `vulnerability-archive.md` 写入“评级复核”小节，并在 `outputs/vulnerability-archive.json` 写入 `rating_review` 字段。`rating_review` 至少包含：

```json
{
  "standard_file": "agent/skills/漏洞评级.md",
  "actual_harm": "已证明/未证明的实际危害",
  "false_positive_filter": "是否命中无实际危害、弱证据、预期公开、不予奖励或过强前置条件",
  "business_impact": "对核心业务、真实用户权益、资产安全、数据机密性/完整性/可用性的影响",
  "evidence_closure": "请求、响应、状态码、关键字段、对照结果和复现条件是否闭合",
  "severity_basis": "对应本文件高危/中危/低危的具体条款",
  "not_higher_reason": "为什么不是更高等级",
  "not_lower_reason": "为什么不是更低等级",
  "final_severity": "high|medium|low|info|undetermined",
  "review_result": "confirmed|downgraded|blocked_need_material|false_positive|no_impact"
}
```

强制复核顺序：

1. 先执行“评级前置误报过滤规则”和“无实际危害判定规则”。
2. 再检查“不予奖励的漏洞类型”。
3. 再判断证据是否闭合到真实业务影响，而不是只停留在版本、标题、状态码、配置、公开页面或理论链路。
4. 再按高危、中危、低危条款定级。
5. 最后写明为什么不是更高等级、为什么不是更低等级。

如果证据不足以回答上述问题，必须先补证据；不能补证据时必须降级或标记为受限未验证。

评级前置误报过滤规则：

所有漏洞评级、漏洞归档、风险总账、阶段性报告和正式报告定级时，都必须优先参考本规则，先判断问题是否对被测目标或厂商本身具有实际危害和实际安全意义，避免把无害现象、理论风险、弱证据风险、预期公开内容或无实际利用价值的发现误报为漏洞。无法证明会影响厂商核心业务、真实用户权益、资产安全、数据机密性/完整性/可用性的问题，不应进入有效漏洞评级；

无实际危害判定规则：

如果一个发现只能证明配置异常、公开暴露、理论可利用、前置条件或候选链路，但无法闭合到厂商核心业务、真实用户权益、资产安全、数据机密性/完整性/可用性的实际损害，则不计入有效漏洞。此类发现最多按信息项、低危观察、受限未验证候选或加固建议记录，不应计入中危/高危成果或专属厂商奖励。

符合以下任一情况时，默认判定为“无实际危害或证据不足”，除非补充证据能闭合实际影响：

1.  仅证明存在某种现象，但未证明可导致账号接管、越权读取、权限提升、业务篡改、资金/订单/后台数据泄露、服务不可用或其它真实安全损失。

2.  利用依赖过强外部前置条件，例如需要第三方日志后台权限、用户社工配合、攻击者已持有有效 token、测试账号、厂商侧回调日志、额外授权写入窗口或业务侧人工确认；在这些条件不存在时，单点本身不能独立造成危害。

3.  证据停留在弱证据或理论风险，例如固定 state、宽松 CORS、危险 HTTP 方法声明、证书签发可能性、token 出现在非敏感请求体、公开后台入口、公开 OIDC/Swagger/health/config 元数据、默认错误页、版本号、静态前端配置等，但未证明敏感读取、写入、认证绕过或业务影响。

4.  访问到的内容属于公开页面、公开附件、公开正文、公开配置、公开健康检查、公开错误提示或预期可访问资源，且无法证明其超出公开目的并造成真实安全风险。

5.  只能作为组合链材料，必须依赖 XSS、账号泄露、子域接管、低权限账号、可回滚测试数据、可写授权或其它尚未成立的前置漏洞才能放大影响；单点本身不应直接定为中危或高危。

不予奖励的漏洞类型

信息泄露类

1. 不涉及敏感信息的CORS配置错误

2. 无意义的内存泄漏、日志泄露、内网IP泄露等（不涉及敏感信息）

3. 示例文件泄露

4. 前端源码泄露

功能缺陷类

5. 地图Key及其他Key泄露造成的资源不受限调用问题

6. 虚假的前端绕过（如通过卡包方式绕过但功能无法实际使用）

7. 上传PDF文件造成的XSS问题

配置缺陷类

8. Web安全配置缺陷或安全加固缺失类漏洞（如缺少X-Frame-Options响应头等）

权限问题类

9. 无意义的未授权接口访问（不涉及敏感信息、无法调用敏感接口）

其他情况

10. 无意义的并发问题（如不涉及利益的点赞、收藏、转发等）

11．仅有文字描述，理论猜测存在，没有完整复现细节的报告会忽略处理。

# 漏洞评级标准

高危：

1.  直接获取服务器权限的漏洞。包括但不限于任意命令执行、上传webshell、任意代码执行。

2.  直接导致严重的信息泄漏漏洞。包括但不限于大量个人敏感信息泄露漏洞。

3.  支付类型逻辑漏洞。包括但不限于影响盈利的修改金额逻辑漏洞。

4.  能直接盗取用户身份信息的影响严重的漏洞。可获取大量敏感信息或者执行命令的SQL漏洞。

5.  直接导致严重影响的逻辑漏洞。可重置管理员的密码，且导致大量敏感信息泄露漏洞。

中危：

1.  能直接盗取用户身份信息的漏洞。包括但不限于一般的SQL注入漏洞（需证明可获取数据库名/用户名）。

2.  直接导致相对严重影响的逻辑漏洞。包括无法导致大量敏感信息泄露的管理员用户的密码重置或普通用户密码的重置。

3.  需交互才能获取用户身份信息的漏洞。包括但不限于存储型XSS漏洞。

4.  任意文件操作漏洞。包括但不限于任意文件读、写、删除、下载等操作。

5.  越权访问。包括但不限于绕过认证访问后台，绕过限制修改用户资料、执行用户操作。

6.  比较严重的信息泄漏漏洞。包含敏感信息文件泄露（如DB连接密码）。

7.  弱口令。但不包括前台弱口令

低危：

1.  普通逻辑漏洞。包括但不限于redis未授权、heapdump敏感信息泄露、短信轰炸等。

2.  需交互才能获取用户身份信息并且有一定利用难度的漏洞。

所有的能定级的漏洞均需要漏洞危害说明、详细复现步骤、完整的POC、关键步骤
