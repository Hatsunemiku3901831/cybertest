````markdown
# 授权安全测试 Agent Skill：可控风险证明与动态信息收集

## 1. Skill 目标

本 Skill 用于让 AI Agent 在**明确授权**的安全测试场景中，自主完成从信息收集、攻击面建模、风险验证、动态重规划，到最终形成“具备获取 shell / 系统控制风险”的证明链。

本 Skill 的最终目标不是实际获取 shell，而是：

```text
证明目标系统已经具备被获取 shell 或被系统级控制的风险条件，
并在真正获取 shell、执行持久化、提权或横向移动之前停止。
````

例外：当漏洞赏金平台政策、专项授权书或用户补充授权明确允许 L5 级别的持久化、横向移动或后渗透链路验证时，可进入本 Skill 的“增强模式”。增强模式不是默认模式，必须先记录授权原文、测试资产、账号边界、停止点和回滚/清理方案。

换句话说：

```text
到最后一道门前停下。
```

---

## 2. 适用范围

本 Skill 仅适用于以下场景：

* 企业授权渗透测试
* 内部安全评估
* SRC / Bug Bounty 授权范围
* 靶场 / CTF / 实验环境
* 蓝队暴露面验证
* 红队前置风险验证
* 攻击路径可达性证明



增强模式例外：

* 仅在 L5 明确授权时，允许对授权范围内的持久化、横向移动和后渗透链路做低影响、可回滚、证据化的最小证明。
* 不允许将 L5 理解为解除边界；未列入平台 scope、授权书或用户补充授权的资产、账号、网段、第三方系统和真实用户数据仍然禁止触碰。

## 3. 核心原则

### 3.0 经验固化入口

执行授权 Web/API 渗透测试时，先套用 `agent/skills/security-testing.md` 的“近期跨案例经验固化”规则，用于前端 bundle 优先级、A/B 越权矩阵、对象存储/Git/公开附件最小闭环、FOFA/nmap/CORS/fallback 误报过滤，以及高危候选与已确认漏洞的边界判断。

### 3.1 信息收集永不停止

信息收集不是一个独立阶段，而是贯穿整个测试流程。

任何新发现都可能成为下一轮信息收集的种子。

**推荐起步动作：** 授权测试任务开始后，立即运行扫描管线建立信息收集基线。若任务是赏金/SRC/补天或用户要求高危优先，则该动作是质量门禁；除非平台规则、目标形式或工具缺失不允许，否则不得跳过：

```bash
# 首次侦察（~15-30 分钟）
./tool/scan_pipeline.py --authorized --domain example.com --mode quick

# 正式渗透（~1-3 小时，完整覆盖）
./tool/scan_pipeline.py --authorized --domain example.com --mode full
```

扫描管线自动编排 subfinder → dnsx → httpx → TLS/端口 → katana/history → gf-patterns → nuclei → ffuf → quality-gate → candidate_queue，产出归一化 JSON 和赏金候选队列。详见 `agent/AGENT.md` 常用命令区的 `scan_pipeline.py` 条目。

quick/full/deep 模式会自动运行 `candidate_queue` 阶段。若从历史任务、手工脚本、Burp 导出或外部扫描结果续跑，应手动运行 `tool/bounty_candidate_queue.py --task-dir ...` 离线生成 `bounty-candidates.json` 和 `bounty-candidates.md`，再根据 P0/P1 攻击队列决定是直接进入定向验证还是升级到 `--mode full`/`--mode deep`。如果只运行了 GF 阶段，也可先读取 GF `result.json` 中的 `top_candidates` 作为临时线索，但最终赏金任务应维护统一候选队列。

例如：

* 域名
* 子域名
* IP
* Host / SNI
* API 路径
* 网关路由
* 内部服务名
* 用户 ID
* tenant ID
* JWT 字段
* 错误信息
* Swagger / OpenAPI
* JS 文件
* source map
* VHost / Host-SNI 差异
* 重定向链
* 目录爆破命中
* object storage URL
* 内网 IP
* Kubernetes service 名称
* Spring Cloud 服务名
* Nacos / Eureka 痕迹
* 容器镜像名
* 版本号
* 业务功能名

Agent 必须持续问自己：

```text
这个新信息能不能改变攻击面图谱？
这个新信息能不能触发第二轮信息收集？
这个新信息是否说明某个信任边界存在问题？
这个低危或中危发现能否和已有发现组合成更高影响的攻击链？
```

---

### 3.2 证据驱动

Agent 不允许基于猜测下结论。

所有判断必须来自：

* 请求
* 响应
* 状态码
* Header
* JSON 字段
* 页面内容
* 错误栈
* 日志片段
* 授权范围内的扫描结果
* 用户提供的业务上下文

允许的表达：

```text
证据显示
疑似
可能存在
可以推断
当前观察到
在授权条件下可验证
```

禁止的表达：

```text
已经拿下
一定存在 RCE
一定能 shell
确定能内网横向
已经控制服务器
```

除非已经有明确证据。

---

### 3.3 最小影响验证

所有测试必须遵循：

```text
只验证必要事实，不扩大影响。
```

允许：

* GET
* HEAD
* OPTIONS
* 低频请求
* 授权账号下的权限边界验证
* 只读接口验证
* 错误信息分析
* Header 差异分析
* 路由差异分析
* Host / SNI 差异分析
* 公开 JS / API 文档分析
* 最小样本越权验证

禁止：

* 高并发 fuzz
* 爆破
* 撞库
* 修改业务数据
* 删除数据
* 触发真实任务
* 造成拒绝服务

## 4. 成功标准

本 Skill 的成功标准不是“拿到 shell”。

成功标准是：

```text
构建一条高可信度证据链，
证明当前系统已经具备被获取 shell 或系统级控制的风险条件，
并在真正控制系统之前停止。
```

示例：

| 风险类型    | 允许证明到哪一步            | 禁止继续做什么            |
| ------- | ------------------- | ------------------ |
| 命令执行风险  | 证明存在可控执行原语          | 不弹 shell，不执行系统探索命令 |
| 文件写入风险  | 证明可写入无害标记文件         | 不写 webshell，不写启动项  |
| 模板注入风险  | 证明表达式可被解析           | 不执行系统命令            |
| 反序列化风险  | 证明危险链路可达            | 不执行 payload        |
| 上传执行风险  | 证明上传目录可写、可访问、存在执行风险 | 不上传可执行后门           |
| 管理后台未授权 | 证明可访问高权限功能          | 不修改配置，不新增账号        |
| 云权限泄露   | 证明权限具备高危动作能力        | 不创建资源，不删除资源        |
| 容器控制风险  | 证明控制面可达或权限异常        | 不进入容器，不执行命令        |
| 网关绕过    | 证明可绕过前置鉴权到达后端       | 不继续利用后端漏洞          |

## 5. 风险可达性等级

Agent 应将攻击链推进程度分为以下等级。

### R0：仅发现公开入口

示例：

```text
发现域名、IP、CDN、Nginx、公开页面。
```

说明：

```text
只是普通暴露面，不构成控制风险。
```

---

### R1：确认业务入口

示例：

```text
公网 Nginx 可转发到 Java Gateway。
公开 API 可正常返回业务 JSON。
```

说明：

```text
确认存在真实业务系统。
```

---

### R2：确认内部架构痕迹

示例：

```text
错误信息泄露内部服务名。
响应中出现 upstream、serviceId、routeId、tenantId、internal host。
```

说明：

```text
可以开始第二轮信息收集。
```

---

### R3：确认访问控制边界异常

示例：

```text
未认证请求触发内部服务路由。
低权限账号访问到高权限字段。
A 用户可读取 B 用户对象。
```

说明：

```text
存在可验证的权限边界问题。
```

---

### R4：确认高危控制原语

示例：

```text
存在任意文件写入风险。
存在可控代码加载风险。
存在可控表达式执行风险。
存在高危管理接口未授权。
存在可影响执行环境的配置写入风险。
```

说明：

```text
已经接近系统控制风险。
必须启动 Critical Boundary Rule。
```

---

### R5：实际系统控制

示例：

```text
获取 shell。
执行系统命令。
提权。
横向移动。
持久化。
```

说明：

```text
禁止进入 R5。
Agent 必须在 R4 停止。
```

L5 增强模式例外：

```text
只有在漏洞赏金平台政策、专项授权书或用户补充授权明确允许 L5 时，
才可进入受控 R5 子集。
受控 R5 仅用于最小化证明持久化、横向移动或后渗透链路，
不得扩大到未授权资产、真实用户数据、隐蔽控制或破坏性操作。
```

---

## 6. Critical Boundary Rule：最后大门停止规则

当 Agent 已经确认以下任意条件时，必须停止继续深入：

* 已确认命令执行能力
* 已确认任意文件写入到敏感路径
* 已确认代码加载点可控
* 已确认可影响服务启动配置
* 已确认可访问高危管理面
* 已确认可修改网关路由
* 已确认可修改任务调度配置
* 已确认可影响 CI/CD
* 已确认可访问容器控制面
* 已确认可访问云控制面高危权限
* 已确认可绕过鉴权访问高危内部 API
* 已确认具备进一步拿 shell 的必要前置条件

停止后，Agent 必须执行：

```text
1. 保存最小必要证据
2. 记录请求与响应摘要
3. 标注风险等级
4. 更新任务内全量风险清单
5. 说明与其它发现的关联和可组合攻击链
6. 说明距离 shell 或更高影响的剩余条件
7. 给出修复建议
8. 不执行最终控制动作
```

### 6.1 增强模式：L5 授权覆盖规则

仅当授权原文明确允许持久化、横向移动或后渗透链路验证时，Critical Boundary Rule 可被 L5 增强模式覆盖。覆盖不是取消停止规则，而是把停止点从 R4 前置条件推进到“已完成一个低影响、可回滚、证据化的最小 R5 proof”。

进入 L5 前必须记录：

```text
L5 明确授权原文：
允许动作：持久化/横向移动/后渗透链路
目标资产和排除资产：
测试账号和测试数据：
允许访问的隔离网段/下游系统：
禁止事项：
停止点：
回滚/清理方案：
紧急停止联系人或平台规则依据：
```

L5 允许：

* 使用测试账号、测试主机、测试对象、测试 secret 或无害命令证明链路。
* 创建可识别、可清理、无隐蔽性的临时持久化 proof。
* 在明确授权的下游资产间验证凭据复用、服务信任、控制面可达性和权限边界。
* 在证明最高影响后立即停止、固化证据并清理或记录待清理项。

L5 仍然禁止：

* 对未列入授权范围的资产、账号、网段、第三方系统或真实用户数据扩展。
* 部署隐蔽后门、规避检测、关闭安全产品、清理安全日志或维持未声明控制。
* 批量导出数据、破坏业务数据、造成拒绝服务或将凭据用于授权链路以外系统。
* 在已证明影响后继续扩大控制面。



## 7. 总体流程

```text
初始化授权范围
    ↓
被动信息收集
    ↓
低影响主动验证
    ↓
构建攻击面图谱
    ↓
关联后台完整 Web/API 测试
    ↓
发现新信息
    ↓
二轮信息收集
    ↓
验证访问控制边界
    ↓
确认高危控制原语
    ↓
到最后一道门前停止
    ↓
如存在 L5 明确授权，进入受控 R5 最小 proof
    ↓
输出证据链与修复建议
```

Agent 应收集：

* DNS 记录
* 真实解析基线：默认解析、可信 TCP DNS/DoH 解析、CNAME/A、是否命中 `198.18.0.0/15` Fake-IP
* 子域名
* TLS 证书
* 证书 SAN
* HTTP Header
* Server Header
* Set-Cookie
* CSP
* robots.txt
* sitemap.xml
* JS 文件
* source map
* 静态资源域名
* CDN 特征
* WAF 特征
* 开放 API 文档
* 页面注释
* 前端路由
* 第三方服务引用

Agent 应提取：

* API 前缀
* 业务模块名
* 环境名
* 服务名
* 内部域名
* bucket 名
* 用户字段
* tenant 字段
* token 字段
* 版本信息
* 框架信息

输出：

```text
初始攻击面图谱
初始技术栈判断
初始风险假设
下一步验证计划
```

## 10. 第三阶段：低影响主动验证

Agent 允许执行低影响验证：

* 存活探测
* 路由存在性验证
* Host / SNI 差异验证
* Header 行为差异验证
* 路径规范化差异验证
* 认证与未认证响应对比
* GET / HEAD / OPTIONS 对比
* 错误响应分析
* 网关路由行为分析

Agent 不得执行：

* 注入利用
* 命令执行尝试
* SSRF 利用
* 上传可执行文件
* 修改数据
* 删除数据
* 触发业务任务
* 高速扫描

目标是判断：

```text
请求是否进入网关？
请求是否进入应用？
请求是否进入服务发现？
请求是否进入 downstream？
认证逻辑发生在网关前还是网关后？
不同身份看到的响应是否不同？
```

## 11. 第四阶段：攻击面图谱构建

Agent 必须维护图结构。

### 节点类型

* domain
* ip
* cdn
* nginx
* gateway
* api
* route
* service
* user
* tenant
* jwt
* bucket
* database_hint
* container_hint
* cloud_hint
* admin_surface

### 边类型

* resolves_to
* proxies_to
* routes_to
* leaks
* authenticates_to
* trusts
* references
* belongs_to
* exposes
* depends_on

示例：

```text
api.example.com
  proxies_to -> nginx
nginx
  routes_to -> Java Gateway
Java Gateway
  routes_to -> alabo-tool
Gateway error
  leaks -> internal service name
```

Agent 每次发现新信息后，必须更新图谱。

### DNS / Fake-IP 防误扫规则

当本机开启 TUN、虚拟网卡或代理时，默认 DNS 可能返回 `198.18.x.x` / `198.19.x.x` 等 Fake-IP。公网 Web/API 目标进入主动探测前必须：

* 不把默认 `dig host`、系统 resolver 或 `nmap host` 结果直接当成真实目标。
* 用 `dig +tcp @1.1.1.1 host A`、`dig +tcp @8.8.8.8 host A`、DoH 或 `tool/origin_exposure_probe.py` 交叉确认真实 A/CNAME。
* 在资产图谱中把 Fake-IP 标为 `dns_artifact` 或任务备注，不建立真实 `resolves_to` 关系。
* 端口扫描只扫真实 IP，并使用 `-n` 禁用扫描器 DNS 解析。
* Web/TLS 验证固定真实 IP 并保留 Host/SNI，例如 `curl --resolve host:443:ip https://host/`、`openssl s_client -connect ip:443 -servername host`。
* 出现大量端口全开、全部服务同质响应、异常低延迟或 IP 落在 `198.18.0.0/15` 时，先判定为可能的 TUN/Fake-IP 噪声，完成复核前不得写入漏洞结论。

## 12. 第五阶段：动态二轮信息收集

当出现新信息时，Agent 必须自动触发新的信息收集循环。

### 12.1 发现内部服务名

发现示例：

```text
Unable to find instance for alabo-tool
```

Agent 应做：

* 提取服务名
* 识别命名规则
* 推测相关模块名
* 查找相关 API 前缀
* 查找前端 JS 中是否引用
* 查找 OpenAPI / Swagger 是否出现
* 对比认证与未认证响应
* 判断是否为真实 downstream
* 更新微服务拓扑图

Agent 不得：

* 直接攻击服务发现组件
* 尝试注册恶意服务
* 修改网关路由
* 利用服务发现漏洞

---

### 12.2 发现越权数据

发现示例：

```json
{
  "userId": 10001,
  "tenantId": 3,
  "mobile": "redacted"
}
```

Agent 应做：

* 判断是否来自当前授权身份
* 判断是否跨用户
* 判断是否跨 tenant
* 使用最小样本验证
* 避免批量读取
* 对敏感字段脱敏
* 记录请求差异
* 标记为高风险候选

Agent 不得：

* 批量枚举用户
* 导出数据
* 下载附件
* 查看与证明无关的隐私内容

---

### 12.3 发现 JWT

Agent 应分析：

* issuer
* audience
* subject
* role
* tenant
* environment
* expiration
* scope
* service name
* internal claim

Agent 应判断：

* 是否暴露内部服务名
* 是否存在角色边界
* 是否存在 tenant 边界
* 是否存在网关信任链

Agent 不得：

* 伪造 token 进行未授权操作
* 爆破密钥
* 绕过签名
* 使用泄露凭据扩大权限

---

### 12.4 发现 OpenAPI / Swagger

Agent 应做：

* 解析 API 分组
* 识别高危接口
* 识别管理接口
* 识别文件上传接口
* 识别回调接口
* 识别任务调度接口
* 识别导入导出接口
* 识别鉴权要求
* 对比文档与实际可访问性

Agent 不得：

* 调用写操作接口
* 触发任务
* 修改配置
* 批量读取数据

---

### 12.5 发现对象存储 URL

Agent 应做：

* 判断是否公开可读
* 判断是否泄露 bucket 名
* 判断是否暴露目录索引
* 判断是否包含敏感文件名
* 标记敏感文件下载风险，如备份包、日志、配置、导入导出文件、补丁包、source map、密钥材料和用户数据导出
* 检查下载 URL 是否依赖可控路径、对象 key、临时签名、鉴权 Cookie、Referer 或一次性 token
* 评估该文件线索是否能与越权、网关绕过、弱鉴权、版本漏洞或配置泄露组合成攻击链
* 只读取最小必要元数据

Agent 不得：

* 批量下载
* 删除对象
* 上传对象
* 猜测大量对象名

---

### 12.6 发现内部 IP / 内部域名

Agent 应做：

* 标记为内部资产线索
* 判断是否出现在错误信息、Header、JS、证书中
* 更新拓扑图
* 检查是否存在公网映射

Agent 不得：

* 扫描内网
* 利用 SSRF 打内网
* 横向访问内部服务

---

## 13. 第六阶段：访问控制边界验证

Agent 应重点验证：

* 未登录 vs 已登录
* 普通用户 vs 管理员
* 用户 A vs 用户 B
* tenant A vs tenant B
* Web 端 vs App 端
* 前端权限 vs 后端权限
* 网关权限 vs downstream 权限
* 有 token vs 无 token
* 过期 token vs 有效 token

只允许最小样本验证。

### 越权风险判断

| 现象                      | 风险   |
| ----------------------- | ---- |
| 未登录访问用户数据               | 高危   |
| 普通用户访问管理员数据             | 高危   |
| A 用户读取 B 用户数据           | 高危   |
| tenant A 读取 tenant B 数据 | 高危   |
| 未认证请求进入 downstream      | 中高危  |
| 未认证请求触发服务发现             | 中危   |
| 仅泄露服务名                  | 低到中危 |
| 仅泄露技术栈                  | 信息   |

## 14. 第七阶段：控制风险前置条件验证

Agent 可以判断是否具备 shell 风险，但不能真的获取 shell。

### 14.1 任意文件写入风险

允许证明：

```text
存在可控文件写入能力。
文件路径、文件名、内容部分可控。
```

停止点：

```text
证明写入能力后停止。
```

禁止：

```text
写入 webshell。
写入启动脚本。
写入 SSH key。
写入 cron。
覆盖配置文件。
```

---

### 14.2 代码执行风险

允许证明：

```text
存在代码执行原语。
危险函数、模板、表达式、插件加载点可达。
```

停止点：

```text
证明执行链路可达后停止。
```

禁止：

```text
执行系统命令。
弹 shell。
建立反连。
读取系统敏感文件。
```

---

### 14.3 管理面未授权风险

探测到与授权目标存在业务、域名、证书、前端路由、接口引用、同组织资产或用户指定关系的后台界面、管理入口、运营后台、管理 API、调试管理面或内置产品管理页时，Agent 必须将其作为关联后台资产执行完整 Web/API 渗透测试流程，而不是只做登录页简查。

关联后台完整流程至少包括：

```text
入口和技术栈识别。
登录/认证流程闭环。
后台页面、API、参数、角色和对象枚举。
未授权访问和垂直越权验证。
配置、文件、上传、导入导出、任务调度入口验证。
验证码、找回密码、OAuth、二维码登录链路验证。
CVE 适用性、风险组合和证据化报告。
```

边界：

```text
不做批量爆破。
不做密码喷洒。
不做短信/邮件轰炸。
不做破坏性写入。
不修改配置。
不新增账号。
不执行真实管理操作。
```

允许证明：

```text
可访问高权限管理页面或高危配置接口。
```

停止点：

```text
访问到高危功能入口后停止。
```

禁止：

```text
修改配置。
新增账号。
上传插件。
执行任务。
导出数据。
```

---

### 14.4 网关 / 服务发现风险

允许证明：

```text
未认证请求可进入服务发现逻辑。
可观察内部服务名。
可触发 downstream 路由解析。
```

停止点：

```text
确认路由与服务发现异常后停止。
```

禁止：

```text
注册服务。
修改路由。
攻击 Nacos / Eureka。
打 downstream 漏洞。
```

---

### 14.5 云权限风险

允许证明：

```text
当前凭据具备高危权限。
```

停止点：

```text
权限枚举完成后停止。
```

禁止：

```text
创建实例。
删除资源。
修改 IAM。
创建访问密钥。
读取大规模数据。
```

## 15. 决策引擎

Agent 每一轮都应执行以下逻辑：

```text
1. 当前发现属于哪个风险等级？
2. 是否产生了新的信息收集种子？
3. 是否改变了攻击面图谱？
4. 是否触及 Critical Boundary Rule？
5. 是否已写入任务内全量风险清单？
6. 是否需要进入正式报告或阶段性报告？
7. 是否能与已有低危/中危/信息泄露组合成更高影响攻击链？
8. 是否出现敏感文件下载风险？
9. 是否出现需要 SMS 验证码的流程，且需标记供后续验证码复用验证？
10. 是否探测到关联后台界面、管理入口、运营后台、管理 API、调试管理面或内置产品管理页？如是，是否已纳入完整 Web/API 渗透测试流程？
11. 关联后台是否已经覆盖认证闭环、后台页面/API/参数枚举、未授权访问、垂直越权、配置/文件/上传/导入导出/任务调度入口、验证码/找回密码/OAuth/二维码链路和 CVE 适用性？
12. 是否遇到登录、注册、找回密码、重置密码、验证码、OAuth/二维码登录或需要登录才能继续观察的场景？如是，是否已用默认测试字段完成一次低频登录/认证流程闭环？
13. 该认证流程是否暴露验证码明文传输、验证码复用、验证码未绑定会话/账号、账号枚举、找回密码 token 泄露、登录态提前下发、前端校验绕过、OAuth state/二维码 ticket 重放等攻击面？
14. 如果尚未发现高危或严重级别漏洞/风险，资产信息收集是否已经完善？
15. 高收益方向是否都已尝试？
16. 是否还有未做的高收益动作或未尝试方向？
17. 如果准备提前结束，阻塞点、缺失信息、人工配合事项和下一步方向是否已明确？
18. 是否需要停止？
19. 是否可以用更低影响方式证明？
20. 是否存在越界风险？
```

### 15.1 登录/认证流程闭环

遇到登录、注册、找回密码、重置密码、验证码、短信/邮箱校验、OAuth/二维码登录或需要登录才能继续观察的场景时，Agent 应做一次完整的低频流程闭环。

默认测试字段：

```text
手机号：13333333333
邮箱：114514@gmail.com
密码：12345678Abc
```

闭环必须记录：

* 前端页面、JS 和接口路径。
* 请求方法、路径、参数、状态码、响应体关键字段。
* `Set-Cookie`、认证 Header、验证码 ID/key/token、跳转位置。
* 成功/失败提示是否区分账号存在性。
* 验证码是否明文回传、可复用、跨会话/跨账号使用、只由前端校验。
* 是否登录态提前下发，或在验证码/二次校验完成前已经获得有效 session。
* 是否需要人工接收验证码、提供测试手机号/邮箱或低权限账号。

边界：

* 只做最小必要样本。
* 不做批量爆破、短信/邮件轰炸、密码喷洒或导致真实账号锁定的重复尝试。
* 如继续验证会触发真实短信/邮件或影响真实账号，记录为人工配合事项并停止扩大。

## 16. 路径优先级规则

### 提高优先级

出现以下情况时，提高路径优先级：

* 内部服务名泄露
* 网关路由差异
* 未认证与认证响应不同
* 用户间响应不同
* tenant 边界异常
* Stack Trace
* OpenAPI 泄露
* Swagger 泄露
* source map 泄露
* 敏感文件下载入口
* 需要 SMS 验证码的高价值业务动作
* 管理接口痕迹
* 文件上传接口
* 任务调度接口
* 配置接口
* 回调接口
* 导入导出接口
* 对象存储 URL
* 云凭据痕迹

---

### 降低优先级

出现以下情况时，降低路径优先级：

* 连续一致 404
* 连续一致 403
* 默认站点
* sinkhole 行为
* 无状态差异
* 无 Header 差异
* 无认证差异
* 超出授权范围
* 可能产生破坏性影响

## 19. 报告结论模板

### 19.1 尚未到达 downstream

```text
当前已确认公网入口可到达 API Gateway，并观察到内部服务路由或服务发现痕迹。
但尚未确认请求进入具体 downstream 服务实例。
当前风险主要是内部架构信息泄露与潜在未认证路由暴露。
尚不足以证明具备获取 shell 能力。
```

---

### 19.2 已到达 downstream，但未发现控制风险

```text
当前已确认请求可进入 downstream 服务实例。
但未发现未授权数据访问、管理面开放、文件写入、代码执行或高危配置修改能力。
当前不支持“具备获取 shell 能力”的结论。
```

---

### 19.3 已确认高危控制前置条件

```text
当前已确认存在高危控制原语：
例如任意文件写入、代码执行链路可达、高危管理接口未授权、配置修改能力或云控制面高危权限。

该证据足以证明系统存在进一步获取 shell 或系统级控制的风险。
根据 Critical Boundary Rule，测试已在真正获取 shell 之前停止。
```

---

### 19.4 已触发停止条件

```text
测试已触发 Critical Boundary Rule。

继续测试可能进入实际系统控制、命令执行、持久化、提权或横向移动阶段。
因此 Agent 已停止深入验证，并输出当前证据链与修复建议。
```

## 20. 示例：Gateway 服务名泄露后的动态流程

初始发现：

```text
GET /app/tool/startup
HTTP 503
Unable to find instance for alabo-tool
```

Agent 判断：

```text
已进入 Java Gateway。
已触发服务发现逻辑。
泄露内部服务名 alabo-tool。
未确认进入 downstream 实例。
```

Agent 更新图谱：

```text
api.example.com
  proxies_to -> nginx
nginx
  routes_to -> Java Gateway
Java Gateway
  routes_to -> alabo-tool
Gateway error
  leaks -> alabo-tool
```

Agent 第二轮信息收集：

```text
1. 分析 /app/tool/* 路由前缀
2. 对比认证与未认证响应
3. 检查 JS 是否引用 tool 模块
4. 检查 OpenAPI 是否存在 tool 分组
5. 检查是否有其他类似服务名泄露
6. 判断 alabo-tool 是否恢复实例
7. 如果响应从 503 变为 200/401/403，重新评估风险
```

Agent 当前结论：

```text
当前尚未证明具备获取 shell 能力。
但已证明存在内部服务枚举和潜在未认证路由暴露风险。
```

---

## 21. 示例：越权后的第二轮信息收集

初始发现：

```text
普通用户 A 请求某接口时返回 userId、tenantId、role 字段。
```

Agent 行动：

```text
1. 判断字段是否属于当前用户
2. 使用授权测试账号 B 做最小对照
3. 比较对象 ID 是否可预测
4. 验证是否存在跨用户读取
5. 验证是否存在跨 tenant 读取
6. 只读取最小必要样本
7. 对敏感字段脱敏
```

如果确认越权：

```text
风险等级提升到 High。
进入二轮信息收集。
```

二轮信息收集关注：

```text
1. 是否存在管理员对象
2. 是否存在配置对象
3. 是否存在文件对象
4. 是否存在导出接口
5. 是否存在任务接口
6. 是否存在回调接口
7. 是否可能进一步影响执行环境
```

停止条件：

```text
一旦发现可修改高危配置、可写文件、可触发任务或可影响执行环境，
立即触发 Critical Boundary Rule。
```

## 22. 示例：文件写入风险的停止点

如果 Agent 发现某功能疑似允许写入文件，应只验证到：

```text
可写入无害标记内容。
可确认路径或对象可控。
可确认该写入可能影响服务行为。
```

```text

该文件是否可被解释器执行。
该写入是否可变成持久化。
```

```text
已确认存在任意文件写入或受控文件写入风险。
在特定部署条件下，该风险可能进一步发展为 webshell 或服务控制。
根据停止规则，未进行实际 shell 获取。
```


## 24. 修复建议生成规则

Agent 应根据风险类型输出修复建议。

### 网关与 upstream

* 未认证请求不得进入服务发现逻辑
* 网关统一前置认证
* downstream 不信任单纯来自网关的 Header
* 隐藏内部服务名
* 统一错误响应
* 禁止默认 Host 返回业务页面
* upstream 仅内网可达
* 安全组限制来源
* 管理面禁止公网访问

### 越权

* 服务端强制校验 userId
* 服务端强制校验 tenantId
* 禁止只依赖前端权限
* 所有对象访问必须做 ownership 校验
* 增加审计日志
* 对敏感接口加权限矩阵测试

### 文件写入

* 上传目录禁止脚本执行
* 文件名随机化
* 限制文件类型
* 内容检测
* 存储与执行环境隔离
* 禁止写入 Web 根目录
* 使用对象存储隔离上传文件

### 管理面

* 管理面仅内网/VPN可达
* 强制 MFA
* 强制 RBAC
* 禁止默认密码
* 禁止公网暴露调试接口
* 关闭未使用插件

### 云权限

* 最小权限 IAM
* 轮换泄露密钥
* 禁止长期 AK/SK
* 启用云审计
* 限制高危 API
* 使用临时凭证

## 27. 最终判断句式

### 不具备 shell 风险证明

```text
当前仅确认存在公开业务入口或内部信息泄露。
尚未发现可导致代码执行、文件写入、管理面控制、配置修改或云控制面的证据。
因此当前不支持“具备获取 shell 能力”的结论。
```

### 具备潜在 shell 风险

```text
当前已确认存在高危前置条件。
在特定部署或配置条件下，该问题可能进一步发展为 shell 或系统级控制。
但本轮未执行最终控制动作。
```

### 具备高可信 shell 风险证明

```text
当前已确认存在可直接通向系统控制的关键原语。
证据已足以证明攻击者具备进一步获取 shell 或系统级控制的能力。
根据 Critical Boundary Rule，测试已在真正获取 shell 之前停止。
```

---

## 28. Skill 终止条件

满足任意条件时，Agent 必须停止：

```text
1. 已证明高危控制原语存在
2. 继续测试会进入 shell 阶段
3. 继续测试会造成数据破坏
4. 继续测试会访问大量敏感数据
5. 继续测试会超出授权范围
6. 目标出现异常或不稳定
7. 已有证据足够支撑高危结论
```

L5 增强模式下，以上第 1、2 条可被明确授权覆盖到一个最小 proof；满足以下任一条件仍必须停止：

```text
1. 已完成一个授权的持久化、横向移动或后渗透链路最小证明
2. 动作将超出 L5 授权原文、测试资产、账号边界、隔离网段或停止点
3. 无法确认回滚/清理能力
4. 需要触碰真实用户数据、未授权第三方系统或隐蔽控制
5. 继续推进只会扩大影响而不会增加报告必要证据
```

停止后只允许：

```text
整理证据
生成报告
给出修复建议
制定复测计划
```

---

## 29. Skill 总结

本 Skill 的核心不是“攻击到最后”，而是：

```text
动态发现攻击面，
递归扩展信息收集，
验证信任边界，
证明高危控制风险，
并在真正获取 shell 之前停止。
```

在 L5 增强模式下，本 Skill 的核心是：

```text
在明确授权的赏金或专项范围内，
对持久化、横向移动或后渗透链路做最小 proof，
并在证明影响后立即停止、清理和报告。
```

Agent 的最终职责是：

```text
证明风险真实存在，
降低测试影响，
保护业务系统，
输出可复现、可修复、可审计的安全结论。
```
