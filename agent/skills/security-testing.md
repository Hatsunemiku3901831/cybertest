# Codex Security Testing Skill：授权渗透测试通用漏洞发现链

本 Skill 是授权安全测试的默认总入口。它以补天/SRC/赏金深挖的方法论作为所有授权渗透任务的默认工作流，并吸收 `webskill.md` 的动态信息收集、证据驱动、风险可达性和停止规则；目标不再限定为“证明 shell 风险”。默认目标是：

```text
发现、验证、组合并归档所有具有真实影响的安全风险；
在证明业务影响、安全边界缺陷或高影响前置条件后停止扩大；
只在明确 L5 授权时推进持久化、横向移动或后渗透链路最小 proof。
```

`webskill.md` 和 `bounty-deep-dig.md` 保留为历史兼容文件；新任务默认读取本文件。

## 1. 适用范围与职责

适用于以下授权场景：

- Web/API 渗透测试、资产探测、漏洞验证、配置检查、协议分析。
- SRC、漏洞赏金、补天项目和厂商授权测试。
- 内部安全评估、蓝队暴露面验证、红队前置风险验证。
- 非生产、预发、演练、停服窗口或局域网目标测试。
- 后台、管理面、网关、对象存储、前端 bundle、移动端 API 和业务逻辑测试。

Codex 的职责：

- 优先使用已有 Flow 日志、报告、命令输出和项目文件，避免重复失败路径。
- 自主完成信息收集、攻击面建模、风险验证、动态重规划、证据归档和报告素材整理。
- 需要脚本、解析、依赖、资料或策略时，由 Codex 自己切换到 Coder/Installer/Searcher/Adviser 模式完成。
- 发现任何风险时维护全量风险清单，并持续判断能否与其它低危、中危、信息泄露或配置缺陷组合成更高影响攻击链。

## 2. 通用原则

### 2.1 命令与低影响原则

- 使用非交互参数。
- 给长命令设置合理 timeout。
- `msfconsole` 必须使用 `-q -x "commands; exit"`，不要进入交互模式。
- 记录负面结果，不要把未验证猜测写成漏洞。
- 高风险动作前确认授权范围、影响面、停止点和回滚方式。
- 未经用户明确要求，不做破坏性写入、刷接口、压力测试、绕过限速、持久化植入。
- `GET`、`HEAD`、`OPTIONS`、只读接口、低频请求、少量已知样本和授权账号边界验证优先。
- 写操作只在明确允许且可回滚时验证；生产环境写入必须停在低影响边界。
- 当用户在任务中说明“默认授权”“写入规则”“提供账号可执行副作用验证”或等价表达时，默认允许在提供账号、测试对象和授权范围内执行低频副作用验证，包括短信/邮件触发、导出、建号/创建对象、同步/任务调度、删除测试对象和上传测试材料。执行时必须使用最小次数、可识别测试对象、可回滚方案和完整证据；成功证明一次即停止扩大。
- 账号间越权需要至少两个授权账号；没有第二账号时只列测试计划，不伪造数据。
- 不要暴力枚举大量 ID；使用少量已知样本和对照请求验证即可。

### 2.1.1 副作用写入默认授权规则



```text
对象命名：优先使用 SEC_TEST_ / CODXTEST_ / 用户指定测试前缀。
短信/邮件：只发往测试手机号/邮箱；默认每类最多 2 次，成功一次即停。
导出：只导出测试对象或最小分页；默认 limit <= 10，不做批量导出。
建号/创建对象：只创建测试账号/测试对象；默认每类最多 2 个，并记录清理方式。
同步/任务调度：优先 dry-run、preview、validate-only 或单测试对象；禁止全量生产同步。
删除/禁用：只作用于测试对象、刚创建对象或用户明确指定可删除对象；先证明对象归属和回滚方式。
上传：优先使用合成测试文件或用户提供的专用测试材料；默认单文件 <= 100KB，避免扩散真实个人敏感材料。
停止点：首次证明权限缺陷、业务动作成功、越权可达或服务端进入敏感处理分支后停止扩大。
```

每次执行副作用动作必须记录：

```text
动作类型、账号/角色、对象 ID、请求方法、路径、关键参数、状态码、响应关键字段、
时间、次数预算消耗、是否成功、是否回滚、回滚方式、证据文件路径、后续风险。
```

### 2.2 DNS / Fake-IP 防误扫

本机开启 TUN、虚拟网卡或代理时，系统默认 DNS 可能返回 `198.18.0.0/15` Fake-IP。公网域名目标不得直接使用默认 `dig`、系统 resolver 或 `nmap 域名` 的结果作为扫描依据。

开始公网目标扫描前，先完成真实 DNS 基线：

- 记录默认解析、可信 TCP DNS/DoH 解析、CNAME/A 结果和是否出现 Fake-IP。
- 使用 `dig +tcp @1.1.1.1`、`dig +tcp @8.8.8.8`、DoH 或 `tool/origin_exposure_probe.py` 确认真实 A/CNAME。
- 过滤 `198.18.x.x`、`198.19.x.x`。
- Web/TLS 验证应固定真实 IP 并保留 Host/SNI，例如 `curl --resolve host:443:ip https://host/`、`openssl s_client -connect ip:443 -servername host`。
- 若 nmap 出现 Fake-IP、全端口异常开放或毫秒级同质响应，先按 DNS/Fake-IP 污染处理，不作为漏洞证据。

### 2.3 证据驱动

不基于猜测下结论。所有判断必须来自：

- 请求、响应、状态码、Header、Cookie、JSON 字段。
- 页面内容、错误栈、日志片段、授权范围内扫描结果。
- 前端 JS、source map、OpenAPI/Swagger、业务上下文。
- 认证态和未认证态对照、A/B 账号对照、真实 404/403/401 对照。

允许表达“疑似、可能、当前观察到、在授权条件下可验证”。禁止在证据不足时写“已拿下、一定存在 RCE、一定能 shell、已经控制服务器”。

### 2.4 风险记录与攻击链原则

- 发现的所有风险都要记录，即使是低危、中危、信息泄露、配置缺陷、弱信号或受限未验证线索。
- 不要把每个漏洞当作孤立问题处理；同一任务中应持续维护风险清单和攻击链假设。
- 每条风险都要标记可组合条件，例如登录态、低权限账号、对象 ID、文件路径、上传点、下载点、回调地址、验证码、内部服务名或配置泄露。
- 低风险至少写入任务内部归档；中风险及以上写入正式报告或阶段性报告，并保留请求、响应摘要、影响判断和复现条件。
- 高危候选和已确认漏洞必须分开。组件版本命中公告、边界设备疑似受影响、安装器暴露、管理登录面公开可以作为高优先级候选；未证明命令执行、文件读写、账号接管、客户端执行或所有者侧影响前，不写成已确认高危。
- 如果未发现高危或严重级别漏洞/风险就准备结束任务，必须先自检资产信息收集、高收益路径和未尝试方向；若受限不可执行，说明阻塞点、缺失材料、人工配合事项和下一步方向。

## 3. 默认方法论：补天式深度挖掘循环

所有授权渗透任务，无论是 SRC、补天、漏洞赏金、普通 Web/API 渗透、企业授权测试、内网测试还是非生产专项，默认使用同一套补天式深度挖掘流程：

```text
资产面最大化
    ↓
机器可读攻击面基线
    ↓
候选队列生成
    ↓
P0/P1/P2/P3 分流
    ↓
按 ROI 优先消耗高收益路径
    ↓
最小低影响证据闭合
    ↓
影响边界停止、评级复核、归档报告
    ↓
复盘自校准，更新下一轮优先级
```

核心原则：

- 不平均审计所有接口；先大量生成候选，再消耗最高 ROI 候选。
- 不把扫描器输出当漏洞；候选只代表验证优先级，结论必须由请求、响应、状态码、业务字段和影响证明闭合。
- 不因 P0/P1 暂时未闭合就归档失败；缺少账号、验证码、测试对象、AppKey/Secret、测试文件或授权窗口时，标记 `blocked_need_material`。
- 不在未完成基线、候选队列和 P0 队列处理前写“全资产穷尽”“高收益方向耗尽”。
- 低危、信息项和受限线索不孤立处理；持续判断能否与认证、权限、文件、后台、网关、源站、缓存、客户端注入、SSRF、SQLi、VHost、重定向或业务状态组合。

### 3.1 资产面最大化

每个任务都输出并持续维护：

```text
资产清单：
行业和业务假设：
入口类型：
认证状态：
技术栈：
API 前缀：
高价值功能：
疑似后台：
疑似对象存储：
移动端入口：
云配置线索：
供应链/公开仓库线索：
ASN/IP/证书归属：
被动 DNS/历史 URL：
公开文档/员工信息 OSINT：
技术栈深度指纹：
CDN/缓存/WAF 归属：
悬挂 DNS / 可疑 CNAME：
需登录继续的入口：
授权规则限制：
```

优先动作：

- 被动收集 in-scope 子域、证书、ASN/IP 归属、被动 DNS、历史 URL、公开 JS、source map、Swagger/OpenAPI、sitemap、robots。
- 搜索公开 GitHub/GitLab commit、issue、CI/CD 产物、公开文档、员工信息和供应链线索；只做被动信息收集和已知 CVE/泄露匹配。
- 低速确认 Web 存活、标题、TLS、Server、框架、错误页面特征、JS 框架版本、字体/图标资源哈希、网关错误、WAF/CDN/源站线索。
- 提取高价值关键词：`upload`、`import`、`export`、`download`、`preview`、`convert`、`task`、`job`、`callback`、`webhook`、`url`、`fetch`、`render`、`pdf`、`excel`、`attachment`、`admin`、`role`、`permission`、`tenant`、`audit`、`config`、`graphql`、`internal`、`debug`。

### 3.2 候选队列与状态机

所有任务默认维护候选队列，推荐输出到 `outputs/bounty-candidates.json` 和 `outputs/bounty-candidates.md`。虽然文件名沿用 bounty，但语义是“高价值攻击候选队列”，普通渗透同样使用。

候选来源至少覆盖：

```text
子域名 / 存活 Web / 端口服务 / VHost / Host-SNI 差异 / 重定向链 /
目录爆破 / URL 爬取 / 历史 URL / JS chunk / source map /
API base / 前端路由 / Swagger/OpenAPI / Actuator/health/config /
登录和 SSO 配置 / OAuth/OIDC 配置 / 移动端 API /
对象存储链接 / 上传下载导入导出接口 / test/pre/dev/staging 环境
```

候选类型至少覆盖：

- SQLi、SSRF、Open Redirect、IDOR/BOLA、OAuth/OIDC/SAML。
- 文件上传、下载、导入、导出、预览、转换。
- Swagger/OpenAPI/Actuator、OSS/STS/Object Storage、Admin/Management。
- Test/Pre/Dev/Staging、Mobile API/Deep Link、API Gateway/Open Platform。
- VHost/Host-SNI、Directory Brute、JS Attack Surface、Core Business API。

候选状态统一使用：

```text
discovered
triaged
high_value
verifying
blocked_need_material
confirmed
false_positive
downgraded
out_of_scope
no_impact
archived
```

### 3.3 P0/P1/P2/P3 分流

- `P0`：最可能直接产出高危或中危结果。每轮优先消耗 P0。
- `P1`：高价值但需要少量补证、测试账号、对象 ID、AppKey/Secret 或测试文件。
- `P2`：低概率但仍有业务价值，或需要较多上下文才能闭合。
- `P3`：信息项、加固项、低收益方向，只做归档或组合链上下文。

P0/P1 默认包括 SQLi、SSRF、账号接管、OAuth/OIDC/SAML、IDOR/BOLA、文件上传/下载/导入/导出、OSS/STS、API 网关、后台/管理端、VHost/Host-SNI、测试环境连接生产数据、重定向组合认证链、目录爆破命中的高价值入口、JS 还原出的核心业务 API。

快速降级方向包括普通安全头、版本号、公开前端配置、无敏感 health/info、无影响 CORS、默认错误页、普通营销跳转和无业务字段的 401/403/404。

只有当 P0 被确认、误报、降级、出 scope、无影响或进入材料阻塞后，才进入 P1/P2。

### 3.4 ROI、时间盒与自校准

不限时不等于无限投入。每条路径都要用 ROI 决策是否继续：

```text
ROI = 历史高危产出概率 × 当前目标暴露程度 × 影响上限 ÷ 验证复杂度
```

评分参考：

| 因素 | 低 | 中 | 高 |
|---|---|---|---|
| 历史高危产出概率 | 普通静态页 | 常规表单/API | 上传、越权、导出、后台、SSRF、支付/租户、缓存、竞态、GraphQL |
| 暴露程度 | 单点弱信号 | 可访问但需条件 | 多入口、多版本、认证态可复现 |
| 影响上限 | 硬化项 | 单用户/单对象 | 跨用户、跨租户、敏感数据、执行、源站绕过、共享缓存 |
| 验证复杂度 | 需额外授权/账号 | 需少量样本 | 当前可低影响验证 |

建议时间盒：

- 单条弱信号：30-60 分钟无新增证据则降级归档。
- 单条高价值路径：2-4 小时仍缺关键前置条件则记录阻塞并切换。
- 单个资产面：完成威胁假设、端点清单、认证闭环、高价值功能分流后再换面。
- 重新升优先级条件：获得第二账号、验证码、测试数据、平台授权、源码线索、新资产或新接口版本。

每次任务结束把结果写入复盘，并维护类目 accepted、duplicate、N/A、平均验证耗时、误报率和奖励/影响等级。accepted 率高、奖励高、耗时低的类目下次提高权重；duplicate 率高或误报率高的类目下次降权或加强行为证据门禁。

### 3.5 深度挖掘循环

每轮循环都回答：

```text
1. P0 队列里当前最可能出高危/中危的候选是什么？
2. 它能否组合认证、权限、文件、上传、后台、网关、源站、缓存、客户端注入、SSRF、SQLi、VHost、重定向或业务状态？
3. 是否还有未测 sibling host、API version、hidden route、object ID、role boundary、缓存键、解析差异、目录命中或 JS 还原出的旧接口？
4. 当前验证是否会触碰授权或平台红线？
5. 最小低影响证明是什么？
6. 如果不能继续，缺少的是账号、验证码、测试数据、AppKey/Secret、测试文件、报备、授权窗口还是对象样本？
7. 当前候选应进入 confirmed、false_positive、downgraded、blocked_need_material、out_of_scope、no_impact 还是继续 verifying？
```

量化停止信号：

- 连续两轮复盘没有新增资产、端点、权限差异或可组合条件。
- 高价值路径全部卡在同一外部条件，且无法由低影响手段补齐。
- 剩余发现只能形成安全头、版本猜测、普通信息泄露等低收益硬化项。
- 继续验证将触碰授权红线或需要额外报备。

若需要更细的候选评分和闭合流程，读取 `bounty-candidate-triage.md` 和 `bounty-closure-playbooks.md`。

## 4. 通用漏洞发现链

### 4.1 初始化

安全测试任务开始时，先确认或记录：

- 授权范围、目标清单、排除资产、测试窗口、账号边界、禁止事项。
- 是否为赏金/SRC、非生产/预发/停服窗口、局域网、CTF/靶场或常规生产授权测试。
- 是否已有历史任务目录、接手文档、资产清单、漏洞归档、Burp 历史、扫描产物或测试账号。
- 是否需要创建 `tasks/YYYY-MM-DD-HHMM-short-task-name/` 归档目录。
- 实际读取的 skill、HackSkills 子技能和触发原因。

如果信息不足但可以黑盒行动，基于已有信息继续推进；如果继续会触碰授权边界或高风险动作，先列明缺失信息。

### 4.2 机器可读资产基线

授权测试任务开始后，优先建立机器可读资产发现基线。赏金/SRC/补天或高危优先任务中，该动作是质量门禁；除非平台规则、目标形式或工具缺失不允许，否则不得跳过。

```bash
# 首次侦察
./tool/scan_pipeline.py --authorized --domain example.com --mode quick

# 正式渗透 / 完整覆盖
./tool/scan_pipeline.py --authorized --domain example.com --mode full

# SPA/API 网关/大型站点或 P0/P1 线索持续存在时
./tool/scan_pipeline.py --authorized --domain example.com --mode deep
```

若从历史任务、手工脚本、Burp 导出或外部扫描结果续跑，应离线生成统一候选队列：

```bash
./tool/bounty_candidate_queue.py --task-dir tasks/YYYY-MM-DD-HHMM-short-task-name \
  --output-json tasks/YYYY-MM-DD-HHMM-short-task-name/outputs/bounty-candidates.json \
  --output-md tasks/YYYY-MM-DD-HHMM-short-task-name/outputs/bounty-candidates.md
```

不能建立自动基线时，必须在任务日志写明替代的手工输入来源和未覆盖项。

### 4.3 信息收集持续循环

信息收集不是独立阶段，而是贯穿测试全程。任何新发现都可能成为下一轮信息收集种子。

优先收集和持续更新：

- 域名、子域名、IP、Host/SNI、TLS 证书、证书 SAN。
- HTTP Header、Set-Cookie、CSP、robots.txt、sitemap.xml、状态码差异。
- JS 文件、source map、前端路由、API base、对象字段、权限字段。
- API 路径、参数、OpenAPI/Swagger、GraphQL、移动端 API。
- 网关路由、VHost/Host-SNI 差异、重定向链、目录爆破命中。
- 后台、管理面、运营后台、调试面、任务调度、导入导出、上传下载。
- 对象存储 URL、公开附件、Git 暴露、配置文件、日志、备份包。
- 内部服务名、内部域名、内网 IP、Kubernetes service、Spring Cloud/Nacos/Eureka 痕迹。
- 用户 ID、tenant ID、订单 ID、文件 ID、组织 ID、JWT 字段、验证码字段。

每轮都问：

```text
这个新信息能否改变攻击面图谱？
这个新信息能否触发第二轮信息收集？
这个新信息是否说明某个信任边界存在问题？
这个低危或中危发现能否与已有发现组合成更高影响攻击链？
```

### 4.4 认证与权限闭环

遇到登录、注册、找回密码、重置密码、验证码、短信/邮箱校验、OAuth/二维码登录或需要登录才能继续观察的场景时，必须做一次完整的低频认证流程闭环。

默认测试字段：

```text
手机号：13333333333
邮箱：114514@gmail.com
密码：12345678Abc
```

闭环记录：

- 前端页面、JS 和接口路径。
- 请求方法、路径、参数、状态码、响应体关键字段。
- `Set-Cookie`、认证 Header、验证码 ID/key/token、跳转位置。
- 成功/失败提示是否区分账号存在性。
- 验证码是否明文回传、可复用、跨会话/跨账号使用、只由前端校验。
- 是否登录态提前下发，或在验证码/二次校验完成前已经获得有效 session。
- 是否需要人工接收验证码、提供测试手机号/邮箱或低权限账号。

边界：

- 只做最小必要样本。
- 不做批量爆破、短信/邮件轰炸、密码喷洒或导致真实账号锁定的重复尝试。
- 用户未声明默认授权写入时，如继续验证会触发真实短信/邮件或影响真实账号，记录为人工配合事项并停止扩大。
- 用户提供测试手机号/邮箱或测试账号时，可按副作用写入默认授权规则触发低频短信/邮件或账号动作；成功一次即停并记录证据、次数和回滚/清理方式。

### 4.5 API / 对象模型 / 权限边界

对认证态 API 先做健康基线。BOLA/IDOR 使用：

```text
A self -> B self -> A 查 B -> B 查 A -> no-token -> fake-object
```

规则：

- self 基线失败、登录态过期或样本对象为空时，不记录越权阴性。
- 普通用户 401/403 不能代表高权限角色安全。
- 缺少店主、管理员、非空订单、银行卡、实名、券、流水等样本时，应标为 `blocked_need_material`，不是闭合。
- 对象 ID、租户 ID、订单 ID、文件 ID 只用已知样本和少量变体验证。
- 未认证读泄露可以引导邻近写接口验证。用户未声明默认授权写入时，写接口用空体、畸形体、不存在 ID、no-op 请求证明是否进入业务逻辑，不触发真实删除、打印、同步、支付、短信、改密、下单、上传持久化或设备控制。用户已声明默认授权写入且提供账号/测试对象时，可按副作用写入默认授权规则执行低频、可回滚、可审计的真实动作。

### 4.6 关联后台完整测试

探测到与授权目标存在业务、域名、证书、前端路由、接口引用、同组织资产或用户指定关系的后台界面、管理入口、运营后台、管理 API、调试管理面或内置产品管理页时，应作为关联后台资产执行完整 Web/API 渗透测试流程。

至少覆盖：

- 入口、技术栈、真实 IP/Host/SNI、CVE 适用性。
- 认证流程闭环、验证码/找回密码/OAuth/二维码链路。
- 后台页面、API、参数、对象模型枚举。
- 未授权访问、水平越权、垂直越权、BFLA/BOLA。
- 配置、文件、上传、导入导出、任务调度、回调接口。
- 风险组合、影响证明和报告归档。

仍需遵守低影响边界：不做批量爆破、密码喷洒、短信/邮件轰炸、破坏性写入、配置修改、新增账号或真实管理操作。用户已声明默认授权写入且提供账号/测试对象时，可按副作用写入默认授权规则执行低频、可回滚、可审计的测试账号/测试对象操作；命中高危管理功能入口或成功完成一次影响证明后停止扩大。

## 5. 风险可达性等级

风险等级用于描述“证据链距离真实影响有多近”，不替代 `漏洞评级.md` 的最终严重性评级。

### R0：仅发现公开入口

只有公开页面、端口、版本、默认页、登录页、普通 401/403/404、无敏感 health/info。通常作为资产或信息项记录。

### R1：确认业务入口

确认目标存在真实业务功能、API、后台、对象模型、认证流程或可交互入口，但尚未证明安全边界异常。

### R2：确认架构或数据线索

发现可推动下一轮验证的线索，例如 API base、source map、Swagger、对象字段、内部服务名、对象存储、配置片段、回调地址、上传下载接口、验证码字段、JWT 字段。

### R3：确认安全边界异常

出现未认证与认证响应差异、A/B 账号对象差异、tenant 边界异常、后台接口弱鉴权、敏感文件可读、对象存储可列举、Git 可恢复、Open Redirect 进入认证链、CORS 可读认证态数据等。

### R4：确认高影响前置条件

已证明存在接近真实影响的关键前置条件，例如：

- 可读敏感数据、可导出业务对象、可访问管理功能。
- 文件上传已完成接受、存储、取回、业务绑定中的关键环节。
- SSRF 可触达内网、metadata、管理面或云凭据路径的最小证明。
- SQLi 已证明布尔/时间/报错差异并可推断数据访问能力。
- 代码执行、模板注入、反序列化、命令执行等已证明危险 sink 或无害命令边界。
- OAuth/OIDC/SAML、找回密码、验证码、二维码或会话链路已证明账号接管前置条件。

### R5：真实影响闭合

已在授权边界内完成真实影响证明，例如读取测试数据、执行无害命令、完成低影响写入、证明账号接管、证明管理操作可达或完成 L5 最小 proof。R5 只能在明确授权、可回滚、可审计的条件下执行。

## 6. Impact Boundary Rule：影响边界停止规则

默认停止点不是“拿 shell”，而是“足以证明真实影响或高影响前置条件已经成立”。

触发以下任一情况时，应停止扩大影响，固化证据并转入评级、归档和修复建议：

- 已证明认证、授权、租户、对象、文件、数据流或业务状态边界异常。
- 已证明敏感数据、管理功能、对象存储、配置、Git、日志、备份或导出文件可被未授权或越权访问。
- 已证明 SQLi、SSRF、RCE、文件写入、模板注入、反序列化、上传链等高危路径的关键前置条件。
- 继续验证需要真实用户数据、有效生产 ID、真实支付/短信/邮件/设备控制、批量导出、破坏性写入、配置修改、持久化、横向移动或后渗透动作。
- 继续验证需要额外账号、验证码、测试订单、测试运单、测试网点/组织 ID、测试 AppKey/Secret、可回滚测试文件或测试窗口。

停止时必须记录：

```text
已证明事实：
证据摘要：
当前风险可达性等级：
是否需要漏洞评级复核：
继续扩大需要的前置条件：
为什么此处停止：
建议厂商内部验证方式：
修复建议：
```

### L5 授权覆盖规则

仅当授权原文明确允许持久化、横向移动或后渗透链路验证时，Impact Boundary Rule 可被 L5 增强模式覆盖。覆盖不是取消停止规则，而是把停止点推进到“已完成一个低影响、可回滚、证据化的最小 R5 proof”。

进入 L5 前必须记录：

```text
L5 明确授权原文：
允许动作：持久化/横向移动/后渗透链路
目标资产和排除资产：
测试账号和测试数据：
允许访问的隔离网段/下游系统：
禁止事项：
停止点：
回滚/清理方案：
紧急停止联系人或平台规则依据：
```

L5 仍然禁止对未列入授权范围的资产、账号、网段、第三方系统或真实用户数据扩展；禁止隐蔽后门、规避检测、关闭安全产品、清理安全日志、批量导出数据、破坏业务数据或在已证明影响后继续扩大控制面。

## 7. 漏洞类型路由

按观察到的现象选择验证路线，并按需读取 `hack-skill.md` 和 HackSkills 子技能：

- Auth / Session / JWT / OAuth / SSO：登录、注册、找回密码、验证码、二维码、token、state、redirect_uri。
- BOLA / IDOR / BFLA：对象 ID、tenantId、orgId、orderId、fileId、角色边界和管理接口。
- File / Upload / Download：上传、导入、导出、预览、转换、附件、路径参数、source map、备份包。
- SSRF：URL、webhook、远程导入、PDF/截图生成、图片抓取、回调地址。
- SQLi / Injection：报错、布尔差异、时间差异、排序/搜索/筛选参数、GraphQL 批量查询。
- XSS / Client-side：反射、富文本、postMessage、DOM sink、缓存投毒、开放重定向组合认证链。
- Business Logic：支付、优惠券、库存、积分、审批、状态流转、竞态和一次性动作。
- Gateway / VHost / Host-SNI：网关路由、源站、隐藏后台、401/403 绕过、Header 信任边界。
- Object Storage / Git / Public Files：公开列举、敏感前缀、单对象 HEAD、业务关联、Git HEAD/config/index/refs。
- RCE / Shell Risk：命令执行、模板注入、反序列化、表达式注入、文件写入、任务调度、CI/CD、文档转换。

## 8. 非生产高强度 Profile

非生产、预发、演练、停服窗口或局域网目标可按授权条件启用更高强度，但仍由本文件的证据、归档和边界规则约束。

强度概览：

- L1：增强验证。更完整目录/参数 fuzz、模板化扫描、小样本弱口令、小样本越权写入、非破坏性 PoC。
- L2：高强度专项。深度 fuzz、公开 PoC 实测、后端处理器真实触发、SSRF 授权内网段、限速/锁定策略测试。
- L3：破坏边界验证。仅在用户明确说可以暂停、崩溃、重建环境时使用；局域网地址默认进入 L3，除非用户限制强度。
- L4：非生产无限制专项。仅在用户明确点名 L4、无限制强度或等价表述时启用。
- L5：赏金/专项后渗透链路验证。仅在授权原文明示允许持久化、横向移动或后渗透链路验证时启用。

启用前必须记录：

```text
目标范围：
测试窗口：
是否可停服：
是否可重启：
是否有快照/备份：
是否允许写入测试数据：
是否允许删除测试数据：
是否允许触发异常任务：
是否允许崩溃/重启服务：
是否明确授权 L4/无限制强度：
是否明确授权 L5/后渗透链路：
回滚方案和清理责任人：
```

缺少这些信息时，不直接执行高强度动作。

## 9. 近期跨案例经验固化

以下规则来自近期授权 SRC / Web/API 渗透测试复盘，适用于路径选择、误报过滤、证据闭环和停止判断。

优先从公开前端资产入手。对 SPA/Vite/Webpack bundle、selector/config、lazy chunk 做离线抽取，重点识别 API base、业务路径、上传/下载、订单/会员/地址/支付/读者/车辆等对象字段、对象存储资源域、broker/video/camera 等集成配置。前端变量名、secret-like 字符串、sourceMappingURL 或路径本身不是漏洞，只有服务端接受、凭据可用、数据可读或权限边界被证明时才升级。

认证态 API 测试必须先做健康基线。BOLA/IDOR 使用 `A self -> B self -> A 查 B -> B 查 A -> no-token -> fake-object` 矩阵；self 基线失败、登录态过期或样本对象为空时，不记录越权阴性。普通用户 401/403 不能代表高权限角色安全，缺少店主/管理员/非空订单/银行卡/实名/券/流水等样本时，应标为阻塞而不是闭合。

未认证读泄露可以引导邻近写接口验证，但生产写入必须停在低影响边界。用户未声明默认授权写入时，若读接口暴露对象 ID 形状、同步配置、表名、业务 handler 或参数结构，可用空体、畸形体、不存在 ID、no-op 请求证明是否进入业务逻辑；不要使用有效生产 ID，不触发删除、打印、同步、支付、短信、改密、下单、上传持久化或设备控制。用户已声明默认授权写入且提供账号/测试对象时，可使用有效测试对象完成低频、可回滚、可审计的最小副作用验证。业务错误响应可能是鉴权缺口信号，但不等于真实数据已被修改。

对象存储、公开附件和 Git 暴露应使用最小证据闭环。对象存储用“公开列举 + 敏感前缀 + 单对象 HEAD + 业务关联”证明，不下载票据正文、大型归档或继续翻页。公开政务/公共服务附件要区分依法公开与最小化脱敏不足，只记录字段类别、数量级和公开上下文，不输出个人信息正文。Git 暴露用 HEAD/config/index/refs/单个 object 证明可恢复性，不执行完整 clone 或 secret 扫库。

误报过滤要前置。FOFA、nmap、all-open、tcpwrapped、CDN/WAF 默认页、SPA fallback 200、相同 body hash、Fake-IP DNS 都不能直接作为漏洞证据；必须用公网 DoH、固定 Host/SNI、协议级 banner/HTTP 响应、真实 404 对照和 body hash 去重确认。CORS 宽松只有在存在 credentials 且浏览器可读敏感认证态数据时才可升级，否则按配置观察或组合风险处理。开放重定向需要说明是否携带授权码、token 或登录态信任链，不能直接写成会话泄露。

上传链按“接受、存储、取回、绑定、渲染/执行”分段评级，缺少取回或业务绑定时不升级为任意文件上传高危。

## 10. 漏洞评级规则

执行漏洞评级时必须读取并服从 `agent/skills/漏洞评级.md`。本节只保留摘要，不能替代该文件。

发现任何新漏洞、风险候选或准备把弱线索升级为漏洞时，必须立即按 `agent/skills/漏洞评级.md` 做评级复核；未复核前只能记录为候选、信息项或受限未验证，不得写成已确认中危/高危。

强制复核至少回答：

1. 是否具有实际危害。
2. 是否命中“不予奖励/无实际危害/证据不足/预期公开内容/过强前置条件”规则。
3. 证据是否闭合到厂商核心业务、真实用户权益、资产安全、数据机密性、完整性或可用性。
4. 为什么不是更低等级。
5. 为什么不是更高等级。
6. 最终等级对应 `漏洞评级.md` 的哪一条标准。

每条已确认漏洞必须在 `vulnerability-archive.md` 写入评级复核块，并在 `outputs/vulnerability-archive.json` 写入 `rating_review` 字段。缺少以上任一复核项时，不得把条目标记为 `confirmed`，不得写入正式报告或阶段报告的有效漏洞章节。

## 11. 决策引擎

每一轮测试都执行：

```text
1. 当前发现属于哪个风险可达性等级？
2. 是否产生了新的信息收集种子？
3. 是否改变了攻击面图谱？
4. 是否触及 Impact Boundary Rule？
5. 是否已写入任务内全量风险清单？
6. 是否需要进入正式报告或阶段性报告？
7. 是否能与已有低危/中危/信息泄露组合成更高影响攻击链？
8. 是否出现敏感文件下载风险？
9. 是否出现需要 SMS 验证码的流程，且需标记供后续验证码复用验证？
10. 是否探测到关联后台界面、管理入口、运营后台、管理 API、调试管理面或内置产品管理页？
11. 关联后台是否已经覆盖认证闭环、后台页面/API/参数枚举、未授权访问、垂直越权、配置/文件/上传/导入导出/任务调度入口、验证码/找回密码/OAuth/二维码链路和 CVE 适用性？
12. 如果尚未发现高危或严重级别漏洞/风险，资产信息收集是否已经完善？
13. 高收益方向是否都已尝试？
14. 是否还有未做的高收益动作或未尝试方向？
15. 如果准备提前结束，阻塞点、缺失信息、人工配合事项和下一步方向是否已明确？
16. 是否需要停止？
17. 是否可以用更低影响方式证明？
18. 是否存在越界风险？
```

## 12. 路径优先级

提高优先级：

- 内部服务名、网关路由差异、未认证与认证响应不同、用户间响应不同、tenant 边界异常。
- Stack Trace、OpenAPI/Swagger、source map、敏感文件下载入口。
- 需要 SMS 验证码的高价值业务动作、管理接口痕迹、任务调度接口、配置接口、回调接口。
- 文件上传、导入导出、对象存储 URL、云凭据痕迹。
- VHost/Host-SNI 差异、源站线索、后台路径、API version、hidden route。

降低优先级：

- 连续一致 404/403、默认站点、sinkhole 行为。
- 无状态差异、无 Header 差异、无认证差异。
- 超出授权范围、无法闭合业务影响、可能产生破坏性影响。

## 13. 输出格式

每轮阶段性输出建议包含：

```markdown
## 当前阶段

## 已确认事实

## 攻击面图谱变化

## 当前风险可达性等级

## 证据摘要

## 已验证假设

## 已否定假设

## 下一轮信息收集计划

## 是否触发 Impact Boundary Rule

## 是否需要漏洞评级复核

## 是否需要停止

## 登录/认证流程闭环记录

## 未发现高危/严重风险时的结束自检

## 提前结束原因、阻塞点、缺失信息和人工配合事项

## 修复建议
```

```text
标题：
```
